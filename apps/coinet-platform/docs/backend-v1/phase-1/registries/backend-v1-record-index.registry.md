## Naming Conventions

```text
BTAR-NNN-short-slug.md
AFE-NNN-short-slug.md
VSE-NNN-short-slug.md
FRP-NNN-short-slug.md
BSCP-NNN-short-slug.md
SCR-NNN-short-slug.md
ADR-NNN-short-slug.md
UDF-NNN-short-slug.md
```

NNN is zero-padded, sequential per type, and never reused. A blocked or withdrawn record keeps its ID; a reshaped resubmission gets a new ID and back-references the original in its `request_origin` field.

## Indexing Workflow

When a new record is created:

1. Save the record file in the correct `records/` subfolder.
2. Add a row to this index with all required fields.
3. If the record is an exception, also add a row to `backend-v1-exception.registry.md`.
4. If the record is a `TAD-A` BTAR, also add a row to `backend-v1-active-task.registry.md`.
5. If the record is an `SCR` that amends a Plan 1.x document, also update the relevant `backend-v1-in-scope.registry.md` / `backend-v1-deferred.registry.md` / `backend-v1-blocked.registry.md`.

All updates happen in the same work session.

## Discovery Rule

A reviewer or future engineer can find every record in the program by reading this single file. If a record file exists on disk but is not in this index, the index is corrected immediately (the record's discoverability is the index's responsibility).

## Lifecycle Hygiene

- Closed records (`COMPLETED`, `WITHDRAWN`, `EXPIRED`, `SUPERSEDED`) remain in this index. They are not deleted; the index is append-only at the row level.
- Status changes update the `Status` and `Last Updated` columns in place.
